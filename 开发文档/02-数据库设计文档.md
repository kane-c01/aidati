## 一、设计原则

1. **统一规范**:所有表带 `id` (BIGINT 自增主键)、`created_at`、`updated_at`(TIMESTAMP)
2. **软删除**:重要业务表带 `deleted_at`(NULL 表示未删除),便于审计与误删恢复
3. **命名规范**:
   - 表名:小写 + 下划线,**单数**(如 `user`, `book`, `paper`)
   - 字段:小写 + 下划线
   - 索引:`idx_<table>_<col>`,唯一索引 `uk_<table>_<col>`,外键 `fk_<table>_<refTable>`
4. **外键策略**:**逻辑外键为主**(应用层维护),少数关键关系建物理外键
5. **JSON 字段**:用于灵活/不规范数据(如题目选项、AI 元数据),严控范围
6. **时间字段**:统一 UTC 存储,展示时转 Asia/Shanghai
7. **不存敏感明文**:OpenID 哈希后存储;未来支付字段加密(V3)

## 四、关键关系与约束

### 4.1 错题入库逻辑

伪代码:

```ts
async function recordMistake(answer) {
  if (answer.is_correct === 1) return;
  const stemHash = sha256(question.stem + question.type);

  await db.upsert('mistake', {
    user_id, question_id, book_id, stem_hash,
    last_wrong_at: NOW,
    wrong_count: { $inc: 1 },
    consecutive_correct: 0,
    status: 'active',
  }, /* on conflict(user_id, stem_hash) update */);
}
```

### 4.2 出题次数检查(伪代码)

```ts
const today = formatDate(now, 'YYYY-MM-DD');
const limit = computeLimit(user); // 5/10/50 + invite_bonus
const used = await redis.incr(`quota:${user_id}:${today}`);
await redis.expireAt(`quota:${user_id}:${today}`, endOfDay);

if (used > limit) {
  await redis.decr(`quota:${user_id}:${today}`);
  throw new QuotaExceededError();
}

// 异步落库
queue.add('persistQuota', { user_id, date: today, used });
```

### 4.3 注销级联

用户主动注销 → 7 天冷静期 → 期满后:
1. `user.deleted_at = NOW, status = -1, openid 改为伪匿名值`
2. 删除:`photo_set, photo, mistake, paper, answer, favorite, notification`
3. 保留:`moderation_log`(法律要求)、`book`(若用户上传过且已通过的书,所有者改为系统账号)

实现:写一个 `cleanupCanceledUsers` job,每天扫描 `deleted_at < NOW() - 7 day AND status = -1` 的记录。

## 五、索引与性能

### 5.1 慢查询热点(预测)

| 场景 | SQL 大致 | 索引 |
|---|---|---|
| 用户主页错题数 | `SELECT COUNT(*) FROM mistake WHERE user_id=? AND status='active'` | `(user_id, status)` |
| 书库分页 | `SELECT * FROM book WHERE status=1 ORDER BY is_recommended DESC, sort_weight DESC LIMIT ?,?` | `(status, is_recommended, sort_weight)` |
| 待审核列表 | `SELECT * FROM book_upload_request WHERE status='pending' ORDER BY created_at` | `(status, created_at)` |
| 查我的试卷 | `SELECT * FROM paper WHERE user_id=? ORDER BY created_at DESC LIMIT ?,?` | `(user_id, created_at)` |

### 5.2 大表预估

| 表 | 行数估算(MVP 第 6 个月) | 月增 |

| question | 300,000 | 100,000 |

## 六、迁移脚本管理

- 工具:**Prisma Migrate** 或 **TypeORM Migration**(推荐 Prisma,DX 较好)
- 命名:`YYYYMMDDHHMM_<description>.sql`
- 永远**不要手改生产 schema**,所有变更走 PR
- 上线流程:开发分支生成 migration → CI 跑 dry-run → 预发执行 → 生产执行

## 七、备份策略

| 数据 | 备份方式 | 频率 | 保留 |
|---|---|---|---|
| 主库全量 | RDS 全量快照 | 每日 | 7 天 |
| Binlog | RDS Binlog | 实时 | 7 天(支持 PITR) |
| 业务关键表(逻辑) | mysqldump → OSS | 每周 | 90 天 |
| 内容审核日志归档 | 每月归档 | 月初 | 6 个月在线 + 归档存储 ≥ 6 个月 |

恢复演练:每季度执行一次"从快照恢复 → 验证关键查询"的演练。
