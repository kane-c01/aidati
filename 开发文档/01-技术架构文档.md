### 2.1 选型表

| 层 | 技术 | 版本 | 理由 |
|---|---|---|---|
| **小程序前端** | 微信小程序原生 + TypeScript | 基础库 ≥ 3.0 | 官方推荐;减少包体积;V2 可选 Taro 升级 |

## 五、跨切面设计

### 5.1 鉴权与角色

- **登录**:`wx.login` 获取 code → 后端换 openid/unionid → 签发 **JWT**(有效期 7 天)+ **refresh token**(30 天)
- **JWT payload**:`{ user_id, openid, role, exp }`,`role ∈ ['user', 'admin', 'super_admin']`
- **角色守卫**:NestJS `RolesGuard`,接口标注 `@Roles('admin')`
- **敏感操作**:管理员任命、用户封禁需要超管,且需要二次确认(前端弹窗 + 后端校验)

### 5.2 限流策略

| 层 | 策略 |
|---|---|
| Nginx | 单 IP 100 req/min |
| 业务接口 | 单用户 60 req/min(普通接口) |
| 出题接口 | 按 7.1.1 业务规则,Redis `INCRBY` 实现 |
| AI 编排 | 单用户并发 1 个出题任务(防滥用) |
| 上传接口 | 单用户 10 文件/分钟 |

### 5.3 内容安全统一拦截

- **统一中间件**:所有 user-generated content 入口经过 `ContentModerationInterceptor`
- **批量调用**:OCR 后的多页文本合并为一次 `msgSecCheck` 调用,降低 API 成本
- **失败策略**:
  - 微信内容安全 API 故障 → 降级为本地敏感词表(系统设置维护)
  - 重要场景(AI 输出)双校验:微信 + 自研规则

### 5.4 幂等性

| 场景 | 实现 |
|---|---|
| 试卷生成 | `Idempotency-Key: ${user_id}-${book_id}-${config_hash}` 在 Redis 缓存 1 分钟 |
| 提交答卷 | 数据库唯一索引 `UNIQUE(paper_id, user_id, version)` |
| 微信支付(V3) | 微信原生 trade_no |

### 5.5 灰度与开关

- 使用「**特性开关 (Feature Flag)**」模式,关键新功能上线前先内部白名单
- 配置存于 Redis,可热更新
- 关键开关:`enable_v2_pdf_upload`、`enable_ai_grading_appeal`、`current_llm_model`

## 六、性能与扩展

### 6.1 关键性能目标(P95)

| 接口 | 目标 RT |
|---|---|
| 登录 | < 500ms |
| 书库列表 | < 300ms |
| 出题接受响应(返回 paper_id) | < 200ms |
| 出题完成(LLM 输出 → 入库) | < 30s |
| 提交答卷接受 | < 500ms |
| 批改完成 | < 30s |
| 错题本列表 | < 300ms |

### 6.2 扩展策略

- **业务后端**:无状态,横向扩容容易
- **AI 编排**:同样无状态,但需要监控 LLM 配额
- **数据库**:MVP 单实例;DAU 1k+ 后启用主从读写分离;5k+ 考虑分表(按 `user_id` 哈希)
- **Redis**:MVP 单实例;后续 Cluster
- **静态资源**:CDN
