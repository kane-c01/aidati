## 二、编码规范

### 2.1 通用

- TypeScript **strict** 模式;严禁 `any`,需要时用 `unknown` + 类型守卫
- ESLint + Prettier:强制统一(参考下方配置)
- 单文件 ≤ 300 行,函数 ≤ 50 行;超出考虑拆分
- **不写**英文 + 中文混合的标识符;变量名英文,注释中文 OK

### 2.2 命名

| 类型 | 规则 | 示例 |
|---|---|---|
| 文件名 | kebab-case | `paper-config.ts` |
| 类名 / 组件名 | PascalCase | `PaperCard` |
| 变量 / 函数 | camelCase | `getPaperList` |
| 常量 | UPPER_SNAKE_CASE | `MAX_PHOTO_PAGES` |
| 类型 / 接口 | PascalCase + 后缀 `Type/Props` | `PaperType`、`PaperCardProps` |
| 枚举值 | UPPER_SNAKE_CASE | `PaperStatus.GENERATING` |

### 2.3 注释

- **只解释 why 不解释 what**:不要写 `// 获取用户信息` 这种废话
- 复杂业务逻辑(配额计算、错题去重)必须有注释说明
- TODO/FIXME 必须带 issue 链接或人名 + 日期

### 2.4 提交规范(Conventional Commits)

```
<type>(<scope>): <subject>

[optional body]
[optional footer]
```

`type`:`feat | fix | refactor | perf | docs | test | chore | style | ci`

示例:`feat(paper): 出题配置页支持自定义题量`

> 配套 commitlint + husky;不规范无法 commit。

### 2.5 分支策略

- `main`:始终可发布,直接对应生产
- `develop`:集成分支
- `feature/<name>`、`fix/<issue>`、`hotfix/<issue>`
- PR 必经 1 人 review + CI 全绿

## 三、小程序专属规范

### 3.1 页面结构

每个页面 4 个文件:`page.ts | page.json | page.wxml | page.wxss`

```ts
// pages/paper-answer/page.ts
import { observer } from 'mobx-miniprogram-bindings';
import { paperStore } from '../../stores/paper';

Page({
  data: { /* ... */ },
  onLoad(options: { paperId: string }) { /* ... */ },
  // ...
});
```

### 3.2 状态管理(MobX)

- **全局 store**:`user`、`paper`、`photo`、`mistake`、`config`
- **页面本地数据**:用 `Page.data`
- 跨页面共享(出题配置 → 加载 → 答题 → 结果)走 `paperStore`

### 3.3 网络请求

统一封装:

```ts
// services/http.ts(节选)
async function request<T>(opts: RequestOpts): Promise<ApiResponse<T>> {
  const start = Date.now();
  try {
    const res = await wxRequest({
      url: env.API_BASE + opts.url,
      header: {
        Authorization: 'Bearer ' + tokenStore.access,
        'X-Request-Id': uuid(),
        'X-Client-Version': `miniprogram/${env.VERSION}`,
        ...opts.header,
      },
      method: opts.method,
      data: opts.data,
      timeout: opts.timeout ?? 15000,
    });

    if (res.statusCode === 401) {
      await tokenStore.refresh();
      return request(opts);
    }
    if (res.statusCode === 429) throw new RateLimitError(res.data.message);
    if (res.statusCode === 422) throw new ContentBlockedError(res.data.message);
    if (res.data.code !== 0) throw new BizError(res.data.code, res.data.message);
    return res.data;
  } finally {
    track('api_call', { url: opts.url, ms: Date.now() - start });
  }
}
```

### 3.4 文件上传(直传 OSS)

- 永远不要把图片走后端转发,**先拿凭证直传**(对应 API 5.1)
- 上传完成后调 5.2 绑定接口
- 失败重试 3 次,使用指数退避

### 3.5 异步任务轮询

- 出题/批改用「**指数退避轮询**」:首次 1.5s,失败 +0.5s,封顶 3s
- 同时设置 60 秒整体超时,显示「AI 繁忙,请重试」

```ts
async function pollPaperReady(paperId: string) {
  const start = Date.now();
  let interval = 1500;
  while (Date.now() - start < 60_000) {
    const res = await paperApi.get(paperId);
    if (res.paper.status === 'ready') return res.paper;
    if (res.paper.status === 'failed') throw new Error('生成失败');
    await sleep(interval);
    interval = Math.min(interval + 500, 3000);
  }
  throw new Error('生成超时');
}
```

### 3.6 用户体验通用项

- **骨架屏**:列表加载用骨架,不用「加载中」转圈
- **空状态**:每个列表必须有 EmptyState
- **错误反馈**:Toast(轻)、Modal(强)、页面级错误屏(致命)
- **网络断开**:全局 `wx.onNetworkStatusChange`,顶部显示「网络异常」
- **下拉刷新 / 上拉加载**:列表页统一支持
- **返回拦截**:答题页未提交时确认弹窗(防误退)

### 3.7 包体积管理

- 主包 ≤ 1.5MB(留 0.5M 空间防意外)
- **分包**:`pages/admin/` 整个 V2 管理工作台放分包
- 大图走 CDN,本地仅存 icons
- 字体不打包,继承系统字体

### 3.8 性能

- 列表用 `recycle-view` 或 `<scroll-view>` + 节流
- `setData` 字段最小化,避免传整个对象
- 图片懒加载 `lazy-load`,占位图缓存

### 3.9 隐私与权限

- 任何获取用户信息(头像/昵称/位置/相册)前都要先调 `wx.getSetting`
- 拍照权限被拒,引导用户去设置打开
- 所有用户输入到 LLM 的内容都要先调本地敏感词过滤(降低后端压力)

### 3.10 埋点

事件命名:`<page>_<action>`,如 `home_search_click`、`paper_submit_success`

字段:
```ts
{ event, page, ts, user_id, session_id, ext: { ... } }
```

埋点上报:批量(每 10 条或 5 秒上报一次),弱网时本地排队,最多保留 100 条。

## 四、UI 与设计规范

### 4.1 设计基线

- **基础库**:微信 ≥ 3.0
- **rpx**:所有尺寸用 rpx,设计稿 750 宽
- **安全区**:统一使用 `<safe-area>` 组件(自定义,基于 `wx.getSystemInfoSync`)
- **暗色模式**:V2 支持,用 `theme.json` + CSS 变量

### 4.2 颜色

| 名称 | Hex | 用途 |
|---|---|---|
| primary | #5B5BD6 | 主操作色(柔和紫) |
| primary-hover | #4949C4 | 按下态 |
| success | #1DBF73 | 正确/通过 |
| warning | #F5A623 | 警告 |
| danger | #E14C4C | 错误/驳回 |
| text-primary | #1A1A1A | 主文本 |
| text-secondary | #666666 | 次文本 |
| text-tertiary | #999999 | 辅助文本 |
| border | #EAEAEA | 分割线 |
| background | #F7F8FA | 背景 |

### 4.3 字号

| 用途 | 字号(rpx) |
|---|---|
| 一级标题 | 36 / 加粗 |
| 二级标题 | 32 / 中粗 |
| 正文 | 28 |
| 辅助文字 | 24 |
| 标签 | 22 |

### 4.4 间距

8pt 栅格:8 / 16 / 24 / 32 / 40 / 48 / 64(单位 rpx,*2 即可)

### 4.5 组件原则

- 主操作按钮全屏宽 / 圆角 12rpx;高度 88rpx
- 卡片圆角 16rpx,阴影 `0 2rpx 12rpx rgba(0,0,0,0.04)`
- 列表项分割线在内部,左边距同卡片内边距

### 4.6 文案规范

- 用动词,**避免被动**:「立即出题」而非「试卷可被生成」
- 错误文案:**说明问题 + 给出下一步**
  - ❌ 「服务异常」
  - ✅ 「AI 繁忙,请稍后再试」
- AI 标识固定:「内容由 AI 生成,仅供参考」(必须包含「AI」「仅供参考」字样,合规要求)

## 五、TypeScript 类型管理

### 5.1 自动生成

后端 `openapi.json` 通过 CI 产出,前端用 `openapi-typescript` 生成:

```json
// package.json scripts
"gen:api": "openapi-typescript ../openapi/v1.json -o ./types/api.ts"
```

### 5.2 业务类型

`types/domain.ts` 中维护:

```ts
export type PaperStatus = 'generating' | 'ready' | 'failed' | 'submitted' | 'graded';
export type QuestionType = 'single' | 'multiple' | 'judge' | 'fill' | 'short_answer';
// ...
```

### 5.3 类型与运行时校验

- 入参/出参用 `zod` 做运行时校验
- 关键 API 响应在拿到后再用 zod 解析一次,失败上报 Sentry

## 六、测试规范

### 6.1 测试金字塔

| 层 | 工具 | 范围 |
|---|---|---|
| 单元 | vitest / jest | utils、stores、纯函数 |
| 组件 | @vue/test-utils(管理后台) | 关键组件 |
| 集成 | supertest(后端) | API 端到端 |
| E2E | Playwright(管理后台) | 核心流程 |
| 小程序 | miniprogram-simulate(可选) | 关键页面 |

### 6.2 必测覆盖

- **配额计算**:边界(0/上限/超出)
- **错题去重 hash**:相同题 hash 一致
- **异常路径**:LLM 失败、OCR 失败、内容拦截
- **幂等键**:同 key 重复不创建多份

### 6.3 覆盖率门槛

- 业务逻辑(`services/`、`stores/`、`utils/`):≥ 70% line
- 整体:≥ 50%

## 八、性能与监控

### 8.1 监控指标(前端上报)

- 启动耗时(冷启动 / 热启动)
- 首屏渲染时间
- API P95 / P99
- 关键页面进入率(首页 / 答题 / 结果)

- `App.onError` + `wx.onUnhandledRejection`
- 全部错误带 `request_id`、`user_id`、`page`、`stack`
- 接入 Sentry

### 8.3 灰度发布

- 微信小程序原生支持版本灰度;每次发版先放 10% → 50% → 100%
- 后端 Feature Flag 联动(如 V2 上传功能上线时)
